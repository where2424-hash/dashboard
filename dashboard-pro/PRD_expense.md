## 1\. 背景與目標

### 1.1 問題描述

製作公司內部人員目前使用紙本或分散式電子檔案進行報帳，流程不透明、單據容易遺失、審核等待時間長、跨部門溝通成本高。

v4 補強重點：此系統涉及財務資料、銀行帳戶、附件與審核紀錄，除操作效率外，需強化狀態一致性、資料可追溯性、權限控管與附件安全，避免日後產生重複報帳、流水號衝突、審核紀錄不清或財務數字不同步等問題。

### 1.2 目標

* 提供統一的數位報帳申請入口，支援單筆與批次上傳兩種模式。
* 透過 AI 辨識收據自動填入關鍵欄位，降低人工輸入錯誤。
* 建立清晰的多階段審核流程，提升審核透明度。
* v4 新增：建立可稽核的狀態轉換、Audit Log、通知矩陣與資料同步規則。
* v4 新增：讓專案總覽能清楚呈現預算、已用金額、待審金額與預估剩餘。

核心目標：讓製作公司內部人員從登入到完成報帳任務的時間縮短 40%。

## 3\. 角色定義

| 角色 | 說明 | 備註 |
| --- | --- | --- |
| 管理人 | 建立專案、設定成員與權限 | 可查看專案全部資料；可兼任製片或出納 |
| 製片 | 報帳審核第一關，選定申請類型 | 可兼任管理人；可審核自己的報帳 |
| 出納 | 報帳審核第二關，確認實體單據與出帳 | 可兼任管理人；可與製片為同一人 |
| 一般成員 | 所有加入專案的人，皆可提交報帳申請 | 申請人為狀態描述詞，非角色名稱 |
| 專案收支審核 | 可編輯收入明細與支出預算 | 於專案總覽權限設定區塊指定 |
| 團隊成員維護 | 可加入申請人等一般成員 | 於專案總覽權限設定區塊指定 |

補充規則：

* 同一人可同時擁有多個角色。
* 一個專案可以有多個管理人。
* 製片與出納為指定人員，可於專案總覽的權限設定區塊設定。
* 申請人同時是製片時，可以審核自己的報帳。
* 製片與出納為同一人時，可以連續審兩關，但系統需分別保存製片審核紀錄與出納審核紀錄，不可只記一筆。
* 管理人可查看全部資料。
* 管理人可以兼任製片或出納審核；審核紀錄需記錄當下操作角色。
* 權限變更需寫入 Audit Log。

## 4\. 報帳審核流程

### 4.1 四種申請類型

重要：申請類型由製片審核時選定，申請人送出時不須選擇。出納如發現類型錯誤，須退件回製片重新選定。

| 類型 | 適用情境 |
| --- | --- |
| 墊付款報銷 | 代墊金已支付，事後申報報銷 |
| 墊付款超支 | 實際費用超出墊付金金額，需補報銷申請出帳 |
| 墊付款結餘 | 實際花費低於墊付金額，需退回差額 |
| 未付款 | 費用尚未支付，由公司直接出帳給廠商或個人 |

流程 1：墊付款報銷 / 超支 / 未付款

* 報帳申請
* 審核-製片（選定類型）
* 審核-出納
* 待收實體單據
* 待匯款
* 已結案

流程 2：墊付款結餘

* 報帳申請
* 審核-製片（選定類型）
* 結餘退回（申請人匯款 + 上傳截圖）
* 出納確認
* 已結案

| 已退件 | 申請人 | 修改重送 | 製片審核中 | v4 補強 | 保留原流水號；重新送出產生 PDF v2，不覆蓋舊版。 |
| 已退件 | 申請人 | 取消申請 | 已取消 | v4 建議新增 | 取消後不得再次送出，但保留紀錄。 |
| 取消權限矩陣 | 申請人 | 取消申請 | 已取消 | v4 補強 | 申請人可在未結案狀態主動取消；取消後不可恢復為可編輯。 |
| 任一審核狀態 | 製片 / 出納 / 管理人 | Reassign 指派代審 | 原狀態不變 | 沿用 v3 | 責任歸屬維持原始操作人，被指定者收到通知。 |

### 4.4 退件與 Reassign 規則

退件：

* 退回前一環節原始操作人（Reassign 後退件仍回原始人）。
* 退件須填寫原因（必填）。
* 申請人編輯原單重送，不需新建，保留原流水號。
* 退件後重新送出會產生 PDF v2，不覆蓋舊版 PDF。
* 通知對象：申請人、製片、退件人。

Reassign：

* 任一審核環節可指派給專案成員清單內的其他成員。
* 原審核人保留查看權限，責任歸屬維持在原始操作人。
* 被指定者收到系統推播通知。
* Reassign 不改變申請狀態。
* Reassign 需寫入 Audit Log。

## 5\. 編號規則與資料關係

### 5.1 流水號（單筆單據）

| 狀態 | 格式 | 範例 | 說明 |
| --- | --- | --- | --- |
| 草稿 | `{專案簡碼}#00001` | `MMA#00001` | v4 修改：草稿建立時即產生正式流水號，不再使用 Temp# |
| 正式送出後 | `{專案簡碼}#00001` | `MMA#00001` | 沿用草稿階段已產生的正式流水號 |

規則：

* 專案簡碼：建立專案時由系統自動抓取專案名稱拼音簡碼（例：牧馬專案 A → `MMA`）。
* 序號：五碼，`00001`\~`99999`，同專案內連續累計。
* 每筆收據一個正式流水號，草稿階段即產生，方便申請人整理實體單據。
* 退件後編輯重送，保留原本流水號不變。
* 申請取消不得釋放原流水號。
* 申請人寄送實體單據時需標明流水號並依序排列。

### 5.2 請款單編號（一份報帳申請）

| 格式 | 範例 | 說明 |
| --- | --- | --- |
| `{公司簡碼}-{YYMM}-{序號三碼}` | `MUY-2604-001` | 一份申請含多筆流水號 |

規則：

* 公司簡碼：建立公司時由系統自動產生拼音簡碼（例：牧馬影視 → `MUY`）。
* YYMM：對應申請日期的年月（2026/04 → `2604`）。
* 序號：三碼，`001`\~`999`，全系統跨專案當月流水。
* 多筆草稿合併送出時，產生一個請款單編號，並帶入每筆草稿既有正式流水號。
* 請款單編號在頂部顯示，流水號顯示於費用明細各列。
* 申請送出後系統自動產生 PDF 供出納核對單據用。

### 5.3 併發、跳號與重試機制

* 流水號與請款單編號產生需使用 DB transaction。
* 成功建立草稿或正式申請後才 commit 編號。
* 正式流水號不可重複。
* 若送出或交易失敗，允許跳號。
* 可接受跳號，但需有系統紀錄，便於稽核追蹤。
* 申請取消不得釋放原流水號。
* 重要送出與審核操作需具備防重複提交機制。

### 5.4 請款單與流水號資料關係

| 資料物件 | 關鍵欄位 | 關係 | 說明 |
| --- | --- | --- | --- |
| Company | company_id, company_code, name | Company 1:N Project | 公司簡碼用於請款單編號。 |
| Project | project_id, project_code, company_id | Project 1:N PaymentRequest | 專案簡碼用於單據流水號。 |
| PaymentRequest | payment_request_id, request_no, applicant_id, project_id, status | PaymentRequest 1:N ExpenseItem | 一份請款單，送出時產生請款單編號。 |
| ExpenseItem | expense_item_id, expense_no, payment_request_id, draft_status, amount, category | ExpenseItem N:1 PaymentRequest | 每筆收據一個正式流水號，草稿階段即產生。 |
| Attachment | attachment_id, expense_item_id, file_url, thumbnail_url | Attachment N:1 ExpenseItem | 原檔與縮圖分離存放。 |
| ApprovalRecord | approval_id, payment_request_id, role, actor_id, action, reason | ApprovalRecord N:1 PaymentRequest | 製片與出納紀錄分開保存。 |
| PdfDocument | pdf_id, payment_request_id, version, file_url | PdfDocument N:1 PaymentRequest | 送出產生 v1，退件後重送產生 v2。 |
| AuditLog | audit_log_id, actor_id, action, target_type, target_id | AuditLog N:1 Target | 所有重要操作不可刪除。 |

* 出納審核中：藍色

* 已取消：灰色

操作功能：

* v4 新增：匯出 Excel 需記錄操作人與時間。
* v4 新增：疑似重複單據需在列表或明細頁顯示警示。

### 7.2 報帳申請表單（一頁式，右側 scrollbar）

流程：基本資料確認 → 收據上傳 → AI 辨識 → 報帳明細確認 → 收款帳戶確認 → 確認送出

區塊一：基本資料

| 欄位 | 來源 | 可否修改 |
| --- | --- | --- |
| 申請人 | 登入帳號自動帶入 | 可手動修改 |
| 所屬專案 | 目前選取專案自動帶入 | 可切換下拉 |
| 申請日期 | 今天日期自動帶入 | 可手動修改 |
| 摘要說明 | 空白 | 選填，送出後可於明細頁補填 |

規則：

* 確認後顯示綠色「已確認」標籤，保留「修改」按鈕。
* 確認後頁面自動 scroll 到下一區塊，下方區塊同步解鎖。

區塊二：收據上傳

* 支援拖曳、選取檔案（可多張）、拍照上傳。
* 支援多次批次上傳，每次新增到既有清單，標示「批次 1 / 批次 2」。
* 格式：JPG、PNG、PDF；單檔上限 20MB；最多 50 張。
* 上傳後 AI 自動辨識，顯示整體進度條（完成 N 張 / 辨識中 N 張 / 失敗 N 張）。
* v4 修改：每筆收據建立草稿時即取得正式流水號，不再使用 Temp#。

區塊三：報帳明細確認

* 欄位：流水號、收據縮圖、單據日期、發票號碼、費用分類、金額、摘要。
* 勾選的項目所有必填欄位完整後，送出按鈕才啟用。
* 缺漏欄位時，列首顯示紅色驚嘆號。
* AI 辨識失敗（紅色欄位）不可勾選，強制保留在草稿。
* 每列有「刪除」按鈕可移除該筆。
* 底部：已勾選 N 張 / 存入草稿 N 張 / 已勾選合計 NT$。
* 「全部存為草稿」→ 所有收據存入草稿並跳轉草稿頁面。
* v4 新增：若疑似重複單據，列上需顯示警示，使用者若仍要送出需填寫說明。

AI OCR 信心分數規則：

| Confidence | 系統行為 | UI 呈現 |
| --- | --- | --- |
| \>= 0.90 | 自動填入 | 正常顯示，可送出。 |
| 0.70 - 0.89 | 自動填入但需人工確認 | 欄位顯示需確認標記。 |
| < 0.70 | 不自動填入 | 欄位標紅，需人工補填。 |
| 辨識失敗 | 不填入 | 強制留在草稿，不可勾選送出。 |

重複單據檢查規則：

* 發票號碼相同。
* 金額相同。
* 單據日期相同。
* 上傳檔案 hash 相同。
* 同一申請人、同一專案、摘要或店家資訊高度相似。
* 若疑似重複，系統需顯示可能重複的歷史申請連結。
* 使用者仍可送出疑似重複單據，但必須填寫說明，且審核人需看到警示。

草稿機制：

* 未勾選收據自動存入草稿。
* 草稿使用正式流水號，不再使用 Temp#。
* 進入新增申請頁若有草稿 → 頂部橘色提示橫幅（整塊可點擊跳轉草稿頁）。
* 草稿可在報帳系統總表的「草稿」tab 找到。
* 草稿刪除後附件同步刪除。
* 草稿被帶入正式申請後，原草稿狀態改為 converted。

草稿資料狀態：

| 狀態 | 說明 |
| --- | --- |
| draft | 草稿可編輯，已產生正式流水號。 |
| submitted | 使用者已送出，系統驗證與建立請款單處理中。 |
| converted | 草稿已成功轉入正式請款單明細。 |
| deleted | 使用者刪除草稿，附件同步刪除。 |

多草稿合併送出：

* 使用者選 5 筆草稿送出時，系統會產生一個請款單編號。
* 每筆草稿帶入既有正式流水號。
* 任一筆驗證失敗，整批不可送出。

區塊四：收款帳戶

區塊五：確認送出

### 7.5 專案總覽頁

頁面結構（三個區塊）

區塊一：專案基本資訊

| 欄位 | 說明 | 可編輯角色 |

| 專案簡碼 | 等寬字體，系統自動產生拼音簡碼，可手動覆寫 | 管理人／製片 |
| 開始日期 | 文字輸入 | 管理人／製片 |
| 預計結束日期 | 文字輸入 | 管理人／製片 |
| 負責人 | 文字輸入 | 管理人／製片 |
| 狀態 | 下拉：進行中 ／ 暫停 ／ 已結案 | 管理人／製片 |

操作：管理人／製片點「編輯」進入編輯模式，有「儲存」與「取消」按鈕。

區塊二：費用總覽

頁首顯示：收支審核：`{審核人姓名}`

三欄摘要卡片：

| 欄位 | 大字（主要） | 小字（補充） |
| --- | --- | --- |
| 預估收入 | 應收合計 | 已收金額 ／ 未收金額 |
| 預計支出 | 預算合計 | 已用金額 ／ 待審金額 ／ 預估剩餘 |
| 目前盈餘 | 已收 − 實際已用 | 預估：應收 − 預計支出 |

資料來源與同步規則：

| 欄位 | 來源 | 是否可編輯 | 更新時機 | 計算規則 |
| --- | --- | --- | --- | --- |
| 預估收入 | 收入明細表 | 可編輯 | 儲存收入明細後 | 應收金額合計。 |
| 已收金額 | 收入明細表 | 可編輯 | 收入狀態改為已收並儲存後 | 已收金額合計。 |
| 未收金額 | 收入明細表 | 系統計算 | 收入明細變更後 | 應收金額合計 - 已收金額合計。 |
| 預計支出 | 支出預算表 | 可編輯 | 儲存支出預算後 | 各分類預算合計。 |
| 已用金額 | 報帳系統 | 不可編輯 | 報帳結案或已出帳後 | 已用金額 - 結餘退回金額。 |
| 待審金額 | 報帳系統 | 不可編輯 | 報帳送出、審核、退件、結案時 | 審核中的請款金額，不計入已用。 |
| 預估剩餘 | 系統計算 | 不可編輯 | 預算、已用、待審變更後 | 預算 - 已用 - 待審。 |
| 目前盈餘 | 系統計算 | 不可編輯 | 收入或支出資料變更後 | 已收金額 - 實際已用。 |
| 預估盈餘 | 系統計算 | 不可編輯 | 收入或預算資料變更後 | 應收金額 - 預計支出。 |
| 使用率 | 系統計算 | 不可編輯 | 預算或已用變更後 | 已用金額 / 預算。超支時可超過 100%。 |

補充規則：

* 被退件的報帳不計入已用。
* 出納審核中顯示為待審金額，不直接計入實際已用。
* 已結案才計入實際已用。
* 結餘退回是已用的扣除項，已用的計算方式為：已用金額 - 結餘退回金額。
* 超支時預算使用率可超過 100%，維持現行設計。

收入明細表格：

欄位：項目名稱 ／ 狀態（應收／已收）／ 應收金額 ／ 已收金額 ／ 預計入帳日 ／ 實際入帳日 ／ 備註

合計列：應收金額合計 ／ 已收金額合計

規則：

* 標題列與數字欄位一律靠左對齊。
* 預計入帳日與實際入帳日使用原生日曆選擇器（`input[type="date"]`）。
* 狀態為「已收」時：已收金額自動帶入應收金額；實際入帳日必填，欄位旁顯示紅色 `!`。
* 點「儲存」時驗證：已收狀態若無實際入帳日則阻擋並標紅。
* 支援逐筆還原（undo stack），角標顯示可還原步數；「取消」一次還原至進入編輯前的狀態。
* 可編輯角色：管理人 ／ 收支審核人員。
* 「＋ 新增」可增加收入列；各列有「✕」刪除。

支出明細表格：

欄位：費用分類 ／ 類型（墊付金／現金）／ 預算（可編輯）／ 已用（系統回傳，唯讀）／ 待審 ／ 預估剩餘 ／ 使用率

費用分類（共 9 類）：人員費用 ／ 器材費用 ／ 交通費用 ／ 餐飲費用 ／ 道具費用 ／ 場地費用 ／ 服裝費用 ／ 雜支費用 ／ 其他

規則：

* 每個分類各自顯示墊付金與現金兩列（rowspan 共用分類 badge）。
* 預算欄位：管理人 ／ 收支審核人員可直接點擊修改。

* 預估剩餘 = 預算 - 已用 - 待審；低於 0 顯示紅色。
* 支援與收入明細相同的逐筆還原機制。
* 合計列顯示所有分類的預算合計、已用合計、待審合計、預估剩餘合計。

### 10.2 Audit Log 欄位

| 欄位 | 說明 |
| --- | --- |

| actor_id | 操作者 ID。 |

### 10.3 保存與保護策略

* 保存年限：Audit Log 最低保存 7 年（法遵要求若更高，依公司政策覆蓋）。
* 不可變更策略：Audit Log 採 append-only，禁止更新與硬刪除；僅允許系統管理者執行法規允許的封存。
* 敏感欄位遮罩：匯出或查詢時，銀行帳號、身分證字號、地址等個資需遮罩；僅具授權角色可檢視完整值。
* 匯出保護：Audit Log 匯出需記錄操作者、時間、篩選條件與檔案指紋（hash）。

## 12\. API 介面

### 12.0 公司工作區 API 共通規則

* 所有公司工作區 API 請求需明確綁定公司上下文：
  * 方式 A：request 明確帶入 `company_id`。
  * 方式 B：由 `workspace context` 解析出 `company_id`（session/route/token）。
* 後端必須在授權層驗證 `company_id` 與登入者公司成員關係，防止跨公司資料存取。
* 若 `company_id` 缺失、與 workspace 不一致或無權限，回傳 `403 FORBIDDEN` 或 `401 UNAUTHORIZED`。

## 15\. 多工作區與公司模式

### 15.1 工作區模型

系統採雙工作區：

* 個人工作區（Personal Workspace）：個人草稿、個人申請、個人通知與偏好設定。
* 公司工作區（Company Workspace）：公司底下所有專案、成員、公司設定、稽核資訊。

登入後導流：

1. 個人帳號登入。
2. 顯示工作區選擇器：`個人工作區 / 公司工作區`。
3. 若選公司工作區，需先選擇公司（Company A / Company B ...）。
4. 進入該公司工作區首頁。

### 15.2 導航與資訊架構（IA）

左側導航頂部新增工作區切換器：

* 個人工作區
* 公司 A
* 公司 B

公司工作區專屬導航：

* 公司設定
* 專案總覽（顯示該公司所有專案卡片）
* 報帳系統（公司視角，可按專案篩選）
* 成員管理
* 通知 / 稽核

顯示規則：

* 在個人工作區顯示個人導向頁面。
* 在公司工作區顯示公司導向頁面。
* 相同路由名稱可依 workspace mode 切不同資料來源與 UI。

### 15.3 權限模型（Company + Project 雙層）

公司層級角色（新增）：

* 公司負責人（Company Owner）：可查看/編輯公司設定、管理公司成員、查看公司所有專案。
* 公司管理員（Company Admin）：可管理成員與專案，權限次於負責人。
* 公司成員（Company Member）：可看到被授權的專案與公司資訊（依授權）。

專案層級角色（沿用）：

* 管理人 / 製片 / 出納 / 一般成員 / 收支審核。

授權規則：

* 使用者需同時滿足「公司可見權限」+「專案成員授權」才能查看專案資料。
* Company Owner、Company Admin 預設可見該公司所有專案。
* 除 Company Owner、Company Admin 外，其餘成員均需綁定專案角色後，才可見特定專案。
* 非公司負責人或未被授權者，不可進入公司工作區。
* 公司層權限變更須同步寫入公司級 Audit Log。

### 15.4 公司設定頁規格

區塊一：公司基本資料

* 公司名稱、統一編號、負責人、地址、聯絡資訊、公司狀態。

區塊二：公司成員 / 角色 / 權限設定

* 成員邀請、角色指派、權限變更、停權/移除。

區塊三：公司專案

* 顯示公司底下所有專案卡片。
* 可建立新專案、封存專案、指派專案管理人。

區塊四：公司安全與稽核

* Audit Log。
* 權限變更紀錄。
* 匯出紀錄。
* 公司資料修改紀錄。

### 15.5 建立公司流程（台灣統編）

流程：

1. 使用者點擊「建立公司」。
2. 輸入統一編號（8 碼）並進行台灣統編 checksum 驗證。
3. 系統驗證統編格式與合法性。
4. 系統查詢公司資料來源（外部或內部資料表）。
5. 自動帶入公司名稱、負責人、地址等資料。
6. 使用者確認與補填缺漏欄位。
7. 系統檢查該統編是否已存在：
   * 若不存在：建立公司。
   * 若已存在：送出加入申請，待公司管理者審核。
8. 上傳公司證明文件，完成審核流程後開通公司工作區。

驗證規則（最低要求）：

* 統編需為 8 碼數字。
* 套用台灣統編 checksum 規則。
* 重複統編不得直接建立新公司。

### 15.6 API 與資料模型補充（v4）

新增核心資料物件：

* WorkspaceContext：`personal` 或 `company`。
* CompanyMembership：user 與 company 的關聯與角色。
* CompanyAuditLog：公司層級稽核紀錄。
* CompanyJoinRequest：加入公司申請與審核狀態。

新增 API（建議）：

* `GET /api/workspaces`：取得可用工作區清單。
* `POST /api/workspaces/select`：切換工作區。
* `GET /api/companies`：取得可見公司。
* `POST /api/companies`：建立公司（含統編驗證流程）。
* `POST /api/companies/:id/join-requests`：送出加入公司申請。
* `GET /api/companies/:id/projects`：取得公司所有專案卡片。
* `PATCH /api/companies/:id/members/:memberId/role`：更新公司成員角色。
* `GET /api/companies/:id/audit-logs`：公司稽核紀錄查詢。

## 16\. React Dashboard 重構藍圖（可擴展架構）

此章節定義從 mockup 進化到可維運產品的前端工程藍圖，目標是「專業、完整、可擴展」。

### 16.1 架構目標

* Domain First：以業務領域（expenses、approvals、projects、notifications）切模組。
* 分層清晰：UI 層不直接碰 API，頁面不直接寫業務邏輯。
* 可替換基礎設施：先接 mock API，後續可無痛接正式後端。
* 可觀測：關鍵流程可追蹤、可監控、可回溯。

### 16.2 推薦技術棧

* React + TypeScript + Vite（或 Next.js App Router）。
* React Router（路由與權限守門）。
* React Query（伺服器狀態管理）。
* Zustand（本地互動狀態）。
* React Hook Form + Zod（表單與驗證）。
* Tailwind CSS（或既有 design tokens 系統）。
* Vitest + Testing Library + Playwright（單元/整合/E2E）。

### 16.3 前端目錄與邊界

```text
src/
  app/              # router, providers, layout, bootstrap
  pages/            # route containers (thin layer)
  features/         # domain modules
    expenses/
    approvals/
    drafts/
    projects/
    notifications/
  entities/         # domain models / mappers
  shared/
    ui/             # reusable components
    api/            # apiClient, interceptors, error mapper
    lib/            # utils (date/money/validators)
    config/         # env, feature flags
```

規範：

* `pages` 只能組裝，不承載核心規則。
* `features` 可獨立迭代與測試。
* `entities` 不依賴 UI。
* `shared` 不可包含業務耦合邏輯。

### 16.4 路由與權限

建議路由：

* `/dashboard`
* `/expenses`
* `/expenses/new`
* `/expenses/:requestId`
* `/projects/:projectId/overview`
* `/notifications`

權限策略：

* Route Guard：未登入或無權限直接攔截。
* Action Guard：按鈕操作再做一次權限檢查（例如取消、匯出、Reassign）。
* Role-Aware UI：同頁面依角色顯示不同操作入口。

### 16.5 API 層與錯誤處理

* 單一 `apiClient` 處理 token、timeout、retry、error mapping。
* 全域錯誤碼映射到可讀訊息與 UI 行為（toast + inline error）。
* 送出/審核/匯出需防重複提交（idempotency key）。
* DTO 與 domain model 分離，避免 API schema 汙染 UI 層。

### 16.6 測試與品質門檻

* Unit：狀態轉換、金額計算、驗證規則。
* Integration：feature + API mock（MSW）。
* E2E：新增申請、退件重送、出納核准、專案總覽同步。
* CI 最低門檻：lint、type-check、test、build 全通過才可合併。

### 16.7 部署與 User Trial

* Staging 使用免費平台（Vercel/Netlify）供 user trial。
* README 必含：啟動方式、環境變數、測試帳號、trial 流程。
* trial 最少驗證 8 條路徑（建立、草稿、送出、審核、退件、重送、已結案、總覽同步）。

### 16.8 三階段落地

* Phase A：架構骨架（router/provider/layout/shared ui/api layer）。
* Phase B：核心流程（總表、申請、草稿、審核明細）。
* Phase C：治理能力（通知、Audit Log、匯出、E2E、部署）。
